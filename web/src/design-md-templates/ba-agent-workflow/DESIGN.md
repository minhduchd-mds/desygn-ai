# BA Agent Workflow Design.md

## Overview
BA Agent Workflow is a business-analysis automation workspace that turns scattered product context into structured requirements, workflow scenarios, acceptance criteria, and UI screen plans. The agent does not wait for a complete manual prompt. It collects sources, builds project memory, proposes requirement gaps, and generates reviewed Markdown artifacts for implementation agents.

The product direction combines Arkon-style scoped knowledge management, Archon-style deterministic workflow execution, and Desygn AI style Figma/design handoff. The BA skill focuses on producing clear `.md` documents that replace ad hoc BA descriptions with reusable, inspectable, versioned requirement files.

## Key Characteristics
- Agent-first BA workflow: discover, classify, ask missing questions, draft, review, revise, and export.
- Human approval before any requirement becomes final.
- Markdown-native outputs for PRD, BRD, user stories, acceptance criteria, screen flows, and implementation handoff.
- Source traceability: every requirement should link back to Figma, uploaded docs, chat history, or stakeholder notes.
- Reusable BA skill package that can be applied across SaaS, internal tools, CRM, dashboard, commerce, and AI workspace projects.

- Monospace font: `JetBrains Mono`, `SFMono-Regular`, `Consolas`, monospace for IDs, Markdown paths, workflow YAML, and generated file names.

- Metadata labels should be 12-13px, medium weight, with normal letter spacing.

## Components
### Agent Workflow Rail
- Shows the active BA pipeline: Collect -> Understand -> Model -> Draft -> Review -> Export.
- Each step has status: idle, running, blocked, needs review, done.
- Clicking a step opens its evidence, generated artifacts, and pending decisions.

### Source Inbox
- Accepts Figma scans, uploaded Markdown/ZIP files, stakeholder notes, screenshots, URLs, and pasted text.
- Each source has type, owner, timestamp, confidence, and extraction status.
- Sources can be included or excluded from the next generation run.

### Project Memory Panel
- Stores product domain, personas, glossary, business rules, entities, permissions, integrations, and known decisions.
- Separates confirmed facts from assumptions.
- Shows stale memory warnings when source documents changed.

### Requirement Table
- Columns: ID, title, type, priority, persona, source, status, acceptance coverage.
- Supports filters for missing acceptance criteria, no source, conflict, duplicate, and ready for development.
- Rows open into detailed Markdown requirement blocks.

### Question Queue
- Lists open BA questions generated by the agent.
- Groups questions by blocker level: must answer now, can proceed with assumption, optional refinement.
- Supports answer, defer, convert to assumption, and attach source.

### Screen Map
- Shows 8-10 generated screens as a flow, with entry points, key actions, states, and dependencies.
- Each screen links to related requirements and acceptance criteria.
- Supports Figma mapping when design nodes are available.

### Markdown Artifact Preview
- Tabs for `BRD.md`, `PRD.md`, `USER_STORIES.md`, `ACCEPTANCE_CRITERIA.md`, `SCREEN_FLOW.md`, `DESIGN.md`, and `AGENT_HANDOFF.md`.
- Provides diff view before saving generated updates.
- Copy and download actions are available per artifact.

### Review Gate
- Presents generated changes as approve/reject/edit decisions.
- Requires approval for requirement status changes, final artifacts, or Figma fix recommendations.
- Keeps an audit trail of reviewer, timestamp, and reason.

## Layout
- Desktop: three-column operational workspace with source/navigation on the left, active document or screen map in the center, and agent memory/review panel on the right.
- Tablet: left rail collapses into tabs; review panel moves below document preview.
- Mobile: single-column review flow optimized for approving questions, assumptions, and generated artifacts.
- Keep tables dense but readable; prefer sticky headers and inline status chips.
- Avoid decorative cards around whole page sections. Use cards only for individual repeated requirements, questions, or artifacts.

## BA Skill Workflow
### 1. Collect Context
- Read selected Figma nodes, design-system variables, component inventory, uploaded Markdown, ZIP files, screenshots, and prior chat history.
- Normalize all inputs into a source registry with IDs.
- Mark each source as design, business, technical, stakeholder, or generated.

### 2. Build Project Memory
- Extract domain, goals, personas, entities, permissions, workflows, integrations, constraints, risks, glossary, and decisions.
- Keep confirmed facts separate from assumptions.
- Record evidence links for every extracted fact.

### 3. Detect Gaps
- Find missing actors, unclear business rules, undefined edge cases, missing acceptance criteria, incomplete screen states, and conflicts between sources.
- Score readiness for BA handoff and development handoff separately.

### 4. Ask Targeted Questions
- Generate only questions that change implementation or acceptance.
- Group questions by priority and allow the user to answer later with explicit assumptions.

### 5. Generate BA Markdown
- Produce structured BA files: `BRD.md`, `PRD.md`, `USER_STORIES.md`, `ACCEPTANCE_CRITERIA.md`, `SCREEN_FLOW.md`, and `AGENT_HANDOFF.md`.
- Include IDs, source references, assumptions, dependencies, and open risks.

### 6. Review And Approve
- Show generated changes as a diff.
- User can approve, edit, reject, or mark as assumption.
- Approved content updates project memory and the final Design.md handoff.

### 7. Export For Agents
- Export compact prompts for Codex, Claude Code, Cursor, Windsurf, Figma Make, and GitHub Agentic Workflows.
- Include build order, screen list, components, acceptance checks, and validation commands.

## Markdown File Model
### `BRD.md`
- Business objective
- Stakeholders
- Success metrics
- Scope and out of scope
- Business rules
- Risks and constraints

### `PRD.md`
- Product goals
- Personas
- Functional requirements
- Non-functional requirements
- Release scope
- Dependencies

### `USER_STORIES.md`
- Epic
- Story ID
- Persona
- User need
- Business value
- Priority
- Dependencies

### `ACCEPTANCE_CRITERIA.md`
- Requirement ID
- Scenario
- Given / When / Then
- Edge cases
- Error states
- Test notes

### `SCREEN_FLOW.md`
- Screen inventory
- Navigation paths
- Entry and exit conditions
- Empty, loading, error, and permission states
- Data required per screen

### `AGENT_HANDOFF.md`
- Implementation plan
- Screen build order
- Component map
- API/data assumptions
- Validation checklist
- Known risks

## Sample Screens
### 1. Agent Mission Control
- Purpose: central dashboard for BA automation status.
- Content: workflow progress, readiness score, pending questions, artifact status, recent source changes.
- Primary actions: run analysis, review changes, export handoff.
- States: idle, running, needs review, blocked, ready.

### 2. Source Inbox
- Purpose: collect and classify source materials.
- Content: Figma scans, Markdown files, uploaded ZIPs, screenshots, stakeholder notes.
- Primary actions: upload, include/exclude source, reprocess, open extracted facts.
- States: queued, processing, processed, failed, stale.

### 3. Project Memory
- Purpose: inspect what the agent has learned.
- Content: domain summary, personas, glossary, entities, permissions, business rules, decisions, assumptions.
- Primary actions: approve fact, edit memory, mark assumption, attach evidence.
- States: confirmed, assumption, conflict, needs source.

### 4. Requirement Discovery
- Purpose: turn sources into structured requirements.
- Content: generated requirements table, priority, source references, owner, completeness score.
- Primary actions: merge duplicates, split requirement, change priority, send to review.
- States: draft, reviewed, approved, rejected.

### 5. Question Queue
- Purpose: resolve ambiguity before implementation.
- Content: agent questions grouped by blocker level and linked requirement IDs.
- Primary actions: answer, defer, convert to assumption, request more source.
- States: unanswered, answered, deferred, assumed.

### 6. Workflow Scenario Builder
- Purpose: model user journeys and business flows.
- Content: actor, trigger, happy path, alternate paths, exception paths, business rules.
- Primary actions: add step, generate scenario, validate coverage, export Mermaid.
- States: incomplete, validated, conflict, ready.

### 7. Screen Map Generator
- Purpose: replace BA prose with 8-10 implementation-ready screen definitions.
- Content: screen cards, navigation order, related requirements, component hints, Figma mapping.
- Primary actions: generate screens, reorder, link requirement, open screen detail.
- States: generated, needs review, mapped to Figma, approved.

### 8. Screen Detail Specification
- Purpose: describe one screen deeply enough for a coding agent.
- Content: layout, components, data fields, actions, validation, empty/loading/error states, permissions.
- Primary actions: edit spec, generate acceptance criteria, preview Markdown, approve.
- States: draft, reviewed, approved, implementation-ready.

### 9. Markdown Artifact Review
- Purpose: inspect generated BA files before they become source of truth.
- Content: tabbed previews for BRD, PRD, stories, acceptance criteria, screen flow, Design.md, agent handoff.
- Primary actions: approve diff, reject section, edit inline, download file.
- States: changed, unchanged, conflict, approved.

### 10. Agent Handoff Export
- Purpose: prepare final package for implementation agents.
- Content: selected files, compact prompt, target agent, build order, test checklist, risk summary.
- Primary actions: copy prompt, download package, create Figma frame, start coding workflow.
- States: ready, missing approval, exported, archived.

## Acceptance Criteria
- The agent can generate at least 8 screens from incomplete BA context and clearly mark assumptions.
- Every final requirement has an ID, source reference, priority, and acceptance criteria.
- Every generated screen links to at least one requirement or is marked as exploratory.
- The user can review diffs before generated Markdown replaces saved artifacts.
- The system separates confirmed facts from assumptions and conflicts.
- Exported handoff includes implementation order and validation checklist.

## Do's
- Ask fewer, higher-impact questions.
- Keep BA artifacts Markdown-native and versionable.
- Show source references and confidence for generated requirements.
- Prefer review gates over silent auto-write behavior.
- Generate screen states: empty, loading, error, permission, success, and edge cases.

## Don'ts
- Do not claim permanent model training when only local/project memory is used.
- Do not overwrite approved requirements without diff review.
- Do not generate screens with no requirement trace.
- Do not hide assumptions inside polished prose.
- Do not treat imported documents as equally reliable without source type and date.

## Agent Prompt Examples
- "Analyze the uploaded stakeholder notes and generate PRD.md, USER_STORIES.md, and 10 implementation screens. Mark missing business rules as questions."
- "Compare the current Figma selection with SCREEN_FLOW.md and list missing states before coding starts."
- "Turn these requirements into Given/When/Then acceptance criteria and export an AGENT_HANDOFF.md for Codex."
- "Find conflicts between approved Project Memory and the new uploaded BA document."
